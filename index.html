Step 4: Create Base Agent Class

üìã What You're Doing

Creating a base class that all specialized agents will inherit from. This provides common functionality like calling Claude and standardizing responses.

üí° Why This Matters

DRY (Don't Repeat Yourself) principle in action. Without this, you'd copy-paste the same Claude API code in every agent.

  • Code Reuse: Write the Claude API call logic once, use everywhere
  • Consistency: All agents use the same API configuration and error handling
  • Maintainability: Update API logic in one place, affects all agents
  • Extensibility: Easy to add new agents - just extend the base class
  • OOP Benefits: Polymorphism allows treating all agents uniformly
‚ö†Ô∏è What If You Skip It?

You'll duplicate the Claude API call code in every single agent. When Anthropic updates their API, you'll need to update 6+ files instead of 1. Your agents will have inconsistent configurations. Adding a new agent becomes a 30-minute task instead of 5 minutes.

üö´ Common Mistakes
  • Mistake 1: Making execute() concrete instead of abstract (child classes forget to implement it)
  • Mistake 2: Hardcoding the model name instead of using config.model
  • Mistake 3: Not passing client to child classes correctly (super() not called)
  • Mistake 4: Forgetting to return message.content[0].text (returning entire message object)

Step 6: Create Orchestrator

üìã What You're Doing

Creating the orchestrator class that coordinates all your agents, manages their execution (parallel or sequential), and combines their results.

üí° Why This Matters

The orchestrator is the conductor of your agent orchestra. It brings the whole system together.

  • Coordination: Single point to manage all agents
  • Parallel Execution: Run multiple agents at once for speed
  • Sequential Execution: Run agents in order when they depend on each other
  • Simplicity: User code just calls orchestrator, not individual agents
  • Flexibility: Easy to change which agents run without changing user code
‚ö†Ô∏è What If You Skip It?

Users will manually instantiate each agent, manually run them, manually collect results. No parallelization (everything sequential). Code becomes messy with agent management logic everywhere. You lose the "system" part of multi-agent system.

üö´ Common Mistakes
  • Mistake 1: Not using Promise.all() for parallel execution (running in loop instead)
  • Mistake 2: Hardcoding agent names instead of making them configurable
  • Mistake 3: Not handling errors when one agent fails (no try-catch)
  • Mistake 4: Creating new Anthropic client for each agent (wasteful)
  • Mistake 5: Making runSequential async but not actually awaiting each agent
üíª Code/Commands

Create: src/orchestrator.js

import Anthropic from "@anthropic-ai/sdk";
import { ExplorerAgent } from "./agents/explorer-agent.js";
import { AnalyzerAgent } from "./agents/analyzer-agent.js";
import { ImproverAgent } from "./agents/improver-agent.js";
import { TesterAgent } from "./agents/tester-agent.js";

export class MultiAgentOrchestrator {
  constructor(apiKey, config = {}) {
    this.client = new Anthropic({ apiKey });
    this.agents = {
      explorer: new ExplorerAgent(this.client, config),
      analyzer: new AnalyzerAgent(this.client, config),
      improver: new ImproverAgent(this.client, config),
      tester: new TesterAgent(this.client, config)
    };
  }

  async runParallel(agentNames, input) {
    console.log(`üöÄ Running ${agentNames.length} agents in parallel...`);
    const tasks = agentNames.map(name => this.agents[name].execute(input));
    return await Promise.all(tasks);
  }

  async runSequential(agentNames, input) {
    console.log(`üîÑ Running ${agentNames.length} agents sequentially...`);
    const results = [];
    for (const name of agentNames) {
      results.push(await this.agents[name].execute(input));
    }
    return results;
  }
}

# Backend Project Rules

## Code Style
- Use TypeScript for all new files
- Use async/await, not callbacks or .then()
- Prefer const over let, never use var
- Use descriptive variable names (no single letters except i, j in loops)

## File Organization
- Controllers: Handle HTTP requests/responses only
- Services: Contain business logic
- Models: Database schema definitions only
- Utils: Pure functions, no side effects

## Error Handling
- Always use try-catch in async functions
- Return consistent error format: { success: false, error: message }
- Log errors before sending to client
- Never expose internal errors to API responses

## API Standards
- Use REST conventions (GET, POST, PUT, DELETE)
- Validate all inputs with express-validator
- Return 200 for success, appropriate 4xx/5xx for errors
- Include pagination for list endpoints

## Database
- Use transactions for multi-step operations
- Always sanitize user inputs (prevent SQL injection)
- Index foreign keys and commonly queried fields
- Use migrations for schema changes, never manual ALTER

## Testing
- Write tests for every new feature
- Test happy path AND error cases
- Mock external dependencies (database, APIs)
- Aim for 80%+ code coverage

## Security
- Never log sensitive data (passwords, tokens)
- Use parameterized queries (prevent SQL injection)
- Validate JWT tokens on protected routes
- Sanitize all user inputs

## Never
- Change unrelated files when implementing a feature
- Skip error handling
- Commit console.log statements
- Use any as a TypeScript type
- Store secrets in code (use .env)

# React Frontend Rules

## Code Style
- Use functional components, not class components
- Use TypeScript for all components
- Use named exports, not default exports
- One component per file

## Component Structure
- Props interface at top of file
- Hooks before return statement
- Event handlers as arrow functions
- Return JSX at bottom

## State Management
- Use useState for component-level state
- Use Context for app-level state
- Use React Query for server state
- Keep state close to where it's used

## Performance
- Use React.memo for expensive components
- Use useMemo for expensive calculations
- Use useCallback for functions passed to children
- Lazy load routes with React.lazy()

## Styling
- Use Tailwind CSS classes
- Create reusable style components for common patterns
- Use responsive classes (sm:, md:, lg:)
- Dark mode support via dark: prefix

## Forms
- Use controlled components
- Validate on blur and submit
- Show clear error messages
- Disable submit while loading

## API Integration
- Use React Query for all API calls
- Show loading states
- Handle errors gracefully
- Implement optimistic updates

## Testing
- Test user interactions, not implementation
- Mock API calls
- Test error states
- Use Testing Library queries

## Never
- Mutate state directly
- Use inline styles (use Tailwind)
- Fetch data in useEffect (use React Query)
- Skip error boundaries
- Forget loading states
